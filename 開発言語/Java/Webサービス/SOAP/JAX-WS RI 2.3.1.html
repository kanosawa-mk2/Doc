SOAPとは？

SOAP (Simple Object Access Protocol) は、XMLベースのメッセージプロトコルであり、異なるシステム間でのデータ交換を可能にするために設計されました。
主にWebサービスで使用され、HTTPやSMTPなどの通信プロトコル上でメッセージを送受信するための仕様を提供します。

SOAPのバージョン

SOAPにはいくつかのバージョンが存在し、それぞれのバージョンで機能の追加や仕様の改善が行われています。
主なバージョンは以下のとおりです。

バージョン公開年主な特徴
SOAP 1.01998年非公式バージョン。W3Cの標準ではなく、MicrosoftとIBMによる提案ベースの仕様。
SOAP 1.12000年W3Cによって公開された最初の正式仕様。広く普及したバージョン。
SOAP 1.22003年W3C勧告。エラー処理の改善、HTTPとの統合強化、XML命名空間の変更などの改良。

SOAPバージョンの歴史

1. SOAP 1.0 (1998年)

  • Microsoft、IBM、ユーザー企業の協力によって開発された非公式バージョン。

  • 仕様が正式に確立される前の段階であり、W3Cの標準にはなっていない。

  • SOAPという名称が使われ始めたが、まだ広く認知されていなかった。

2. SOAP 1.1 (2000年)

  • W3C (World Wide Web Consortium) によって公式に公開された最初のバージョン。

  • HTTPを主なトランスポートプロトコルとして採用し、POSTリクエストを使用してXMLメッセージを送信する方式を確立。

  • WSDL (Web Services Description Language) との統合が進められ、SOAP Webサービスの普及が始まる。

  • ただし、以下のような問題が指摘されていた：

    • HTTPとの統合が不完全

    • エラーハンドリングが不明確

    • XMLの冗長性による処理コストの高さ

3. SOAP 1.2 (2003年)

  • W3C勧告として正式に採用されたバージョン。

  • SOAP 1.1 の課題を改善し、より明確な仕様に。

  • 主な改良点：

    • エラーハンドリングの改善

      • Fault要素を標準化し、エラー情報を明確化。

    • HTTPとの統合強化

      • application/soap+xmlという新しいMIMEタイプを定義し、メッセージ識別を容易に。

    • 拡張性の向上

      • モジュール方式の導入により、新機能を容易に追加可能に。

    • XML名前空間の変更

      • http://schemas.xmlsoap.org/soap/envelope/ → http://www.w3.org/2003/05/soap-envelope/

    SOAP 1.2 は現在もWebサービスの標準として利用されているが、RESTful APIの普及により新規導入は減少傾向。

SOAPの現状

1. SOAP 1.1 が広く普及していた

SOAP 1.1 は 2000年に W3C によって公開された最初の正式仕様 であり、多くの企業やシステムで標準的に採用されました。
一方、SOAP 1.2 は 2003年に W3C 勧告となったものの、すぐには広く普及しませんでした。

影響:

  • SOAP 1.1 に基づく既存の Web サービスとの互換性を維持するため、多くの実装（JAX-WS や Metro など）では デフォルトを SOAP 1.1 のままにしている。

2. SOAP 1.2 の仕様変更による互換性の問題

SOAP 1.2 は、いくつかの重要な改善を加えましたが、その結果 SOAP 1.1 との後方互換性が完全ではなくなった ため、デフォルトとして採用しにくくなりました。

主な変更点

項目SOAP 1.1SOAP 1.2
エラー処理 (Fault)<faultcode> <faultstring> などの要素<Code> <Reason> に変更され、エラー構造が変更
HTTP 統合text/xml を使用application/soap+xml に変更され、MIME タイプが異なる
メソッド制限SOAPAction ヘッダが必須SOAPAction は不要になった
拡張性拡張性が低いモジュール化により拡張しやすい

影響:

  • SOAP 1.2 では application/soap+xml を使うため、既存の SOAP 1.1 クライアントとの互換性がなくなる ことがある。

  • SOAPAction ヘッダが不要になった ことで、従来の SOAP 1.1 クライアントが SOAP 1.2 サーバーと通信できない場合がある。

  • エラー処理の仕様が変わった ため、SOAP 1.1 ベースのアプリケーションをそのまま移行すると、予期しない動作になることがある。

→ これらの違いから、デフォルトで SOAP 1.2 にすると既存のシステムが壊れる可能性があるため、引き続き SOAP 1.1 がデフォルトになっている。

3. SOAP 1.2 はそれほど普及しなかった

SOAP 1.2 は技術的には優れているものの、SOAP 1.1 からの移行コストの高さ や REST（RESTful API）の普及 などにより、あまり使われませんでした。

影響:

  • SOAP 1.1 のまま動作しているシステムが多いため、新しいバージョンの JAX-WS / Metro でもデフォルトを変更するメリットが少なかった。

  • SOAP 1.2 をデフォルトにすると、不要なトラブルが増える可能性があった。

4. 仕様として SOAP 1.2 を強制していない

JAX-WS や Metro の仕様では、SOAP 1.1 / 1.2 のどちらもサポートするが、どちらをデフォルトにするかは実装側に委ねられている。
多くの実装では 互換性を重視して SOAP 1.1 をデフォルトにしている。

結論

  • 歴史的な理由: SOAP 1.1 は最初の標準仕様であり、多くのシステムがこれに基づいて構築された。

  • 技術的な理由: SOAP 1.2 は改善点が多いが、SOAP 1.1 との互換性が完全ではなく、移行コストがかかる。

  • 普及の問題: SOAP 1.2 への移行があまり進まなかったため、デフォルトを変更するメリットが少なかった。

  • 互換性維持: 既存の SOAP 1.1 システムをそのまま動かすために、デフォルトを変更せず、必要な場合のみ SOAP 1.2 を明示的に選択する方式を採用。
