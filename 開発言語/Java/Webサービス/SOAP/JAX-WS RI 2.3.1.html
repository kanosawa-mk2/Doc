SOAPバージョンの歴史

1. SOAP 1.0 (1998年)

  • Microsoft、IBM、ユーザー企業の協力によって開発された非公式バージョン。

  • 仕様が正式に確立される前の段階であり、W3Cの標準にはなっていない。

  • SOAPという名称が使われ始めたが、まだ広く認知されていなかった。

2. SOAP 1.1 (2000年)

  • W3C (World Wide Web Consortium) によって公式に公開された最初のバージョン。

  • HTTPを主なトランスポートプロトコルとして採用し、POSTリクエストを使用してXMLメッセージを送信する方式を確立。

  • WSDL (Web Services Description Language) との統合が進められ、SOAP Webサービスの普及が始まる。

  • ただし、以下のような問題が指摘されていた：

    • HTTPとの統合が不完全

    • エラーハンドリングが不明確

    • XMLの冗長性による処理コストの高さ

3. SOAP 1.2 (2003年)

  • W3C勧告として正式に採用されたバージョン。

  • SOAP 1.1 の課題を改善し、より明確な仕様に。

  • 主な改良点：

    • エラーハンドリングの改善

      • Fault要素を標準化し、エラー情報を明確化。

    • HTTPとの統合強化

      • application/soap+xmlという新しいMIMEタイプを定義し、メッセージ識別を容易に。

    • 拡張性の向上

      • モジュール方式の導入により、新機能を容易に追加可能に。

    • XML名前空間の変更

      • http://schemas.xmlsoap.org/soap/envelope/ → http://www.w3.org/2003/05/soap-envelope/

    SOAP 1.2 は現在もWebサービスの標準として利用されているが、RESTful APIの普及により新規導入は減少傾向。

SOAPの現状

あなた:
metorのサンプルを見るとsun-jaxs.xmlというファイルが含まれていました。このファイルはなんですか？
ChatGPT:

sun-jaxws.xml は、JAX-WS (Java API for XML Web Services) を使って サーバーサイドの Web サービスを公開する際に、エンドポイントを構成するためのデプロイメントディスクリプタです。

📌 簡単に言うと：

sun-jaxws.xml は、JAX-WS RI（Reference Implementation）や Eclipse Metro を使った Web アプリケーションにおいて、
どのクラスを Web サービスのエンドポイントとして公開するかを記述する設定ファイルです。

🗂 配置場所

通常は以下に配置されます：

bash
WEB-INF/sun-jaxws.xml

🧾 書式の例

xml
<?xml version="1.0" encoding="UTF-8"?>
<endpoints xmlns="http://java.sun.com/xml/ns/jax-ws/ri/runtime"
           version="2.0">
    <endpoint
        name="HelloService"
        implementation="com.example.HelloServiceImpl"
        url-pattern="/hello" />
</endpoints>

各属性の意味：

属性名説明
name任意のサービス名（識別用）
implementationサービスを実装したクラス（@WebService が付与されている）
url-patternこのエンドポイントにアクセスするためのURLパス

📦 使用される場面

  • JAX-WS RI や Eclipse Metro を Tomcat などのサーブレットコンテナで使う場合、sun-jaxws.xml は どのクラスをサービスとして公開するかを Tomcat に伝えるために必要です。

  • Java EE / Jakarta EE のようにサーバーに組み込み済みの環境では、@WebService アノテーションだけで十分なこともありますが、Tomcat のような軽量サーブレットコンテナではこの設定が必要です。

💡 注意点

  • sun-jaxws.xml は JAX-WS RI/Eclipse Metro に特有のファイルです。Apache CXF など他のスタックでは使用されません。

  • web.xml に JAXWSServlet を定義し、sun-jaxws.xml を読み込ませる形で動作します。

xml
<servlet>
    <servlet-name>jaxws-servlet</servlet-name>
    <servlet-class>com.sun.xml.ws.transport.http.servlet.WSServlet</servlet-class>
    <load-on-startup>1</load-on-startup>
</servlet>

<servlet-mapping>
    <servlet-name>jaxws-servlet</servlet-name>
    <url-pattern>/hello</url-pattern>
</servlet-mapping>

ポイント内容
目的JAX-WS Web サービスのエンドポイント定義
使用環境JAX-WS RI, Eclipse Metro（Tomcatなどの軽量サーバー）

✅ クライアント作成の再生成が「不要」なケース

  • WSDL やスキーマ（XSD）が変更されていない

  • SOAP バージョンやバインディング（document/literalなど）に変更がない

  • 既存クライアントが JAX-WS の標準 API で実装されている

  • Metro 固有の機能（例えば FastInfoset や WSIT 拡張）を使っていないか、ごく標準的に使っている

これらの条件がそろっていれば、クライアントはそのまま動作する可能性が高いです。

⚠️ クライアント再生成を検討すべきケース

次のような場合は、wsimport によるクライアント再生成を検討した方が安全です：

条件理由
Metro 4系で JAX-WS 実装の内部構造が変わっているパッケージ名変更や依存関係の見直しがあるため
使用中の API が非推奨または削除された古いバージョン特有の実装に依存している場合
新しい WS-* 機能（セキュリティ、信頼性など）を使いたい新しいポリシー定義や処理方式が入っていることがある
Java SE 11 以降への移行を同時に行うJAXB や JAX-WS がモジュール外に分離されているため

📦 通信内容（SOAPメッセージ）の互換性について

Metro は 標準の SOAP 1.1 / 1.2 + WSDL 1.1 に準拠しているため、以下の点を守れば通信内容に大きな違いは発生しません：

  • SOAP バージョンが変わらないこと（通常は 1.1）

  • エンコーディングスタイルやバインディングが同じ

  • XML Schema に互換性がある

ただし、FastInfoset や WS-Addressing, WS-Security などを利用している場合は、シリアライゼーションやヘッダ構造に違いが出ることがあります。
